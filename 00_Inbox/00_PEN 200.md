## Tema 9 Application Attacks

1. Directory Traversal (Salto de Directorios)

El objetivo aquí es leer archivos fuera del directorio raíz (`web root`) del servidor.

- **Rutas Relativas vs Absolutas:** Intenta leer archivos usando la ruta completa (ej. `/etc/passwd`) o usando secuencias relativas de escape `../` (en Linux) o `..\` (en Windows). Puedes apilar tantos `../` como quieras; una vez llegues a la raíz `/`, no pasarás de ahí.
- **Evasión de Filtros (URL Encoding):** Si la aplicación bloquea o filtra la secuencia `../`, debes probar codificando los caracteres. Por ejemplo, cambiando el punto por su valor codificado en URL: `%2e%2e/`.
- **💡 Qué probar en la OSCP:**
    - **Identificación:** Vigila cualquier URL que tenga parámetros que carguen archivos, como `?page=admin.php` o `?language=en.html`.
    - **No uses el navegador:** Los navegadores web a menudo intentan normalizar las URL y arruinan los ataques de Traversal. Utiliza siempre **cURL** o el **Repeater de Burp Suite**.
    - **Objetivos en Linux:** Si logras leer `/etc/passwd` y ves usuarios legítimos, tu siguiente paso automático debe ser intentar leer sus claves privadas SSH (ej. `../../../../home/usuario/.ssh/id_rsa`) para ganar acceso al sistema.
    - **Objetivos en Windows:** En Windows no hay archivo `/etc/passwd`. Intenta leer `C:\Windows\System32\drivers\etc\hosts` para verificar la vulnerabilidad. Luego, dependiendo del servidor (ej. si usa IIS), busca archivos sensibles como `C:\inetpub\wwwroot\web.config` o los logs en `C:\inetpub\logs\LogFiles\W3SVC1\`.

2. Vulnerabilidades de Inclusión de Archivos (LFI / RFI)

A diferencia del Directory Traversal que solo _lee_, las vulnerabilidades de inclusión **ejecutan** el archivo en el contexto de la aplicación si este contiene código fuente válido (como PHP).

- **Log Poisoning (Elevación de LFI a RCE):** Si encuentras un LFI, puedes lograr ejecución remota de código (RCE) envenenando un archivo de registro.
    - _Cómo hacerlo:_ Modifica tu cabecera `User-Agent` en Burp Suite insertando un payload PHP como `<?php echo system($_GET['cmd']); ?>`. Luego, usa el LFI para incluir el log de acceso de Apache (ej. `/var/log/apache2/access.log`). Tu código PHP se ejecutará, permitiéndote pasar comandos mediante el parámetro `&cmd=`.
- **PHP Wrappers (Evasión y Ejecución):**
    - **Wrapper Base64 (****php://filter****):** Esta es la técnica que mencionabas. Es vital porque si incluyes un archivo PHP normal a través de LFI, el servidor lo ejecuta y no puedes ver el código fuente original. Al usar `php://filter/convert.base64-encode/resource=admin.php`, el servidor te devuelve el código fuente del archivo codificado en Base64. Al decodificarlo en tu terminal, podrás leer contraseñas conectadas a bases de datos u otra lógica oculta.
    - **Wrapper Data (****data://****):** Si la directiva `allow_url_include` está activada en PHP, puedes inyectar código directamente en la URL sin necesidad de envenenar logs. Ejemplo de ejecución directa codificada: `data://text/plain;base64,PD9waHAgZWNobyBzeXN0ZW0oJF9HRVRbImNtZCJdKTs/Pg==&cmd=ls`.
- **RFI (Remote File Inclusion):** Obliga a la aplicación a descargar y ejecutar una webshell alojada en tu máquina. Levanta un servidor web en tu Kali (`python3 -m http.server 80`) que sirva un archivo como `simple-backdoor.php` y pásalo como parámetro.
- **💡 Qué plantearte en la OSCP:** Pregúntate siempre si el LFI que encontraste tiene acceso a los logs del servidor (Apache/Nginx/IIS) o si los _PHP wrappers_ están habilitados para lograr una reverse shell directa.

3. Vulnerabilidades de Subida de Archivos (File Upload)

Ocurre cuando puedes subir archivos a un servidor y luego acceder a ellos.

- **Archivos Ejecutables y Evasión:** El objetivo es subir una webshell (ej. `.php`). Si la aplicación bloquea la extensión `.php` mediante una "lista negra", intenta evadirla:
    - Cambia mayúsculas/minúsculas: Sube el archivo como `backdoor.pHP`.
    - Usa extensiones antiguas/alternativas de PHP: Prueba con `.phps` o `.php7`.
- **Archivos No Ejecutables (Combinación de Ataques):** Si es absolutamente imposible ejecutar código (por ejemplo, los archivos caen en un lugar donde no se interpretan), combina la subida con Directory Traversal en el nombre del archivo. Usando Burp, cambia el nombre del archivo subido a `../../../../../../../root/.ssh/authorized_keys` y sube tu clave pública SSH para intentar sobrescribir el archivo de claves del administrador y ganar acceso.
- **💡 Qué plantearte en la OSCP:** Si descubres un formulario de carga, sube un archivo legítimo y fíjate exactamente **dónde** se guarda y **cómo** lo nombra el servidor. Si el servidor dice "Ya existe" al subirlo dos veces, úsalo para forzar fuerza bruta y adivinar la estructura de directorios.

4. Inyección de Comandos (OS Command Injection)

Sucede cuando un desarrollador pasa el _input_ del usuario directamente a comandos del sistema operativo en el _backend_ sin sanear.

- **Pruebas de Inyección:** Utiliza caracteres separadores del sistema operativo, como `;` (punto y coma), `&` o `&&` para concatenar tu propio comando al que la aplicación ya está ejecutando.
- **Reconocer el Entorno:** Si estás atacando un Windows, necesitas saber si el comando se está interpretando en `CMD` o en `PowerShell`. Puedes inyectar este test: `(dir 2>&1 *`|echo CMD);&<# rem #>echo PowerShell`. El resultado en pantalla revelará el entorno.
- **Evasión de Filtros:** Si comandos como `ipconfig` son detectados por un filtro, codifica en URL los símbolos especiales (`;` se vuelve `%3B`, `&` se vuelve `%26`). A veces el filtro bloquea comandos específicos pero no los delimitadores.
- **💡 Qué probar en la OSCP:** Una vez que confirmes la ejecución, utiliza herramientas nativas (_Living off the Land_). Si estás en Windows PowerShell, inyecta un comando con `IEX` y un `WebClient` para descargar un script como `powercat.ps1` desde tu Kali y enviarte una reverse shell codificada en base64 para evitar que caracteres malos rompan la inyección
